01 — Stan obecny

Jak działa pamięć w Vanilla OpenClaw

OpenClaw oferuje trzy oficjalne backendy pamięci. Każdy ma swoje ograniczenia — a community szuka czegoś lepszego.

📄

memory-core (domyślny)

Pliki Markdown jako source of truth. MEMORY.md + memory/YYYY-MM-DD.md. Semantic search via sqlite-vec. Embeddingi wymagają API key (OpenAI/Gemini/Voyage).

⚠️ MEMORY.md wstrzykiwany W CAŁOŚCI mimo memory_search (#26949)

Działa, ale marnuje tokeny
🗄️

memory-lancedb (plugin)

LanceDB vector store, OpenAI 1536-dim. Auto-capture działa, ale autoRecall domyślnie wyłączony — pamięć się zapisuje, ale nigdy nie wraca.

#25202 — „persistent memory non-functional"

Zepsuty out of the box
🔬

QMD (experimental)

BM25 + vectors + reranking sidecar. Wymaga bun install qmd. Pierwsze wyszukiwanie pobiera modele GGUF.

Puste wyniki

04 — Nasze rozwiązanie

memory-unified — Zunifikowana architektura

Jeden plugin, dwa backendy, trzy hooki. SQLite + FTS5 + HNSW z prawdziwymi embeddingami. Zero zewnętrznych API.

🎯

RAG Slim

Mikro-podsumowania 20-30 tokenów zamiast pełnego tekstu. 90% mniej tokenów na pamięć = więcej miejsca na rozmowę.

🏠

Lokalne embeddingi

Qwen3 na NVIDIA Blackwell (128GB) via Ollama. $0 kosztów API. Brak zależności od OpenAI/Gemini.

⚡

Dynamiczny routing

Z 93 narzędzi do 3-12 per request. 80% mniej narzędzi w kontekście — szybciej, taniej, celniej.

🔄

Auto-capture

Hook after_tool_call loguje każde wywołanie. Nic nie ginie nawet jeśli model zapomni zapisać.
